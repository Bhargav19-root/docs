## Overview

The linkutm API is REST + JSON. All endpoints live under `/api/v1`. JSON in, JSON out. Standard HTTP status codes for errors.

## Base URL

```
https://api.linkutm.com/api/v1
```

For self-hosted or staging environments, replace the host accordingly.

## Authentication

Most endpoints authenticate with a JWT bearer token, issued by the auth flow (login or OTP verification response) and identifying the user. The Links API also accepts a workspace API key. See [Authentication](/api/auth) for the full flow.


```bash
curl https://api.linkutm.com/api/v1/links \
  -H "Authorization: Bearer <jwt_token>" \
  -H "x-workspace-id: <workspace_uuid_or_slug>"
```

<Note>
**Workspace context.** Most endpoints require an `x-workspace-id` header carrying a workspace UUID or slug. For JWT auth, omitting it returns `404` or `400`. For API key auth the header is optional — the key is workspace-scoped so the workspace is injected automatically. If you include the header with an API key, it must match the key's workspace or the request is rejected with `401`.
</Note>

### API keys

Workspaces can generate long-lived API keys (format `lk_live_<hex>`) for server-to-server use. Create and revoke them under **Settings → Developer → API keys** in the dashboard.

An API key authenticates the **Links API**. Send it as the bearer token:

```bash
curl https://api.linkutm.com/api/v1/links \
  -H "Authorization: Bearer lk_live_<hex>"
```

A key is bound to the workspace it was created in, so the `x-workspace-id` header is optional when you authenticate with a key. If you do send it, it must match the key's workspace.

<Note>
An API key covers the core Links endpoints: create, list, read, update, delete, bulk operations, and imports directly under `/links`. It does not cover link sub-resources served by separate controllers (`/links/:id/analytics`, `/links/:linkId/tags`, `/links/:linkId/pixels`), nor other modules (`workspaces`, `folders`, `tags`, `payments`, and so on). Those require a JWT bearer token.
</Note>

## Required headers

| Header | Required | Notes |
|---|---|---|
| `Authorization: Bearer <token>` | Yes | JWT token |
| `x-workspace-id: <uuid_or_slug>` | Yes for workspace-scoped endpoints | UUID or slug |
| `Content-Type: application/json` | For POST/PATCH | |

## Rate limiting

A global rate limit of **100 requests per minute per IP** applies across all endpoints. Exceeding it returns `429 Too Many Requests`. Back off and retry. See [Errors](/api/errors) and [Best practices](/api/best-practices).

## API modules

The `v1` API spans these modules: `links`, `analytics`, `workspaces`, `domains`, `tags`, `folders`, `templates`, `utm-rules`, `parameters`, `pixels`, `api-keys`, `google-analytics`, `storage`, `payments`, and `auth`.

## Versioning

The API is at `v1`. Breaking changes go behind a new version (`v2`). Within `v1`, only additive changes (new optional fields, new endpoints) are made.

## Quick start

<Steps>
  <Step title="Get a token">
    Log in via the auth endpoint or copy a JWT from your browser session.
  </Step>
  <Step title="Find your workspace">
    `GET /workspaces` returns the workspaces you belong to. Note the `id` (UUID) or `slug`.
  </Step>
  <Step title="Create a link">
    ```bash
    curl -X POST https://api.linkutm.com/api/v1/links \
      -H "Authorization: Bearer $TOKEN" \
      -H "x-workspace-id: $WORKSPACE_SLUG" \
      -H "Content-Type: application/json" \
      -d '{
        "url": "https://example.com/landing",
        "utmSource": "google",
        "utmMedium": "cpc",
        "utmCampaign": "q2_launch",
        "customSlug": "q2-launch"
      }'
    ```

    Response: `201 Created` with the full link object.
  </Step>
</Steps>
